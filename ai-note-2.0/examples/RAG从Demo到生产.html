RAG从Demo到生产

鹅厂面试官视角：Demo跑通只是10%的工作
原文来源吴师兄

一、数据准备：知识库质量决定RAG上限

1. 切块不是技术问题，是理解问题

5000份PDF文档（产品说明书、理赔条款、培训资料、历史案例），第一版用PyPDF提取+500字一刀切，结果用户问「重疾险的等待期是多久」，top-3 chunk全是语义碎片——三段都提到「等待期」但无一能完整作答。

核心问题：按字数机械切块会把完整语义切碎，向量检索命中了关键词但信息残缺。

2. 三招修复切块策略

  • 结构化解析——识别标题、段落、表格、列表。标题+下属内容作为一个chunk，表格单独成chunk，语义独立的列表项也单独成chunk。
  • 加上下文路径——每个chunk开头补充层级路径，如产品名称：XX重疾险 > 第三章保险责任 > 3.2等待期规定。即使chunk只有一句话，LLM也能知道它在讲什么。
  • 动态窗口——检索时不只返回命中chunk，还返回前后各一个chunk，补全被切断的语义。
召回准确率从 0.62 → 0.84

二、检索召回：找最有用的，不是最相似的

1. Embedding模型选型

OpenAI text-embedding-ada-002对中文保险条款检索效果差——问「意外伤害包括哪些情况」，召回的是「意外医疗」「意外身故」等相关但不直接的内容。换 BGE-large-zh（智源开源中文模型）后，中文语义理解更细腻，召回精度大幅提升。

2. 召回阈值：一个参数决定生死

阈值召回率说明
0.650.89top-5里有2个噪音
0.720.84top-5基本都有用
0.780.71开始漏掉边缘case
选0.72——宁可少召回，也不要让LLM被噪音带偏。精度优先。

3. 混合检索：向量+关键词缺一不可

纯向量检索的致命缺陷：只看语义相似度，不看关键词匹配。用户问「保单号A12345的理赔进度」，embedding会把ID编码得很模糊，召回一堆通用文档却找不到具体保单。

最终方案：向量检索 + BM25关键词检索 → 合并去重 → Reranker重排。

召回准确率从 0.84 → 0.91

三、Query理解：用户问的不一定是系统该搜的

1. 四种意图类型

  • 事实查询——「重疾险保障范围？」→ 直接检索
  • 计算问题——「投保50万免赔额5000能赔多少？」→ 路由到计算模块
  • 数据库查询——「上个月理赔审批平均时长？」→ 走NL2SQL
  • 带时间约束——「最新车险理赔流程？」→ 检索+时间过滤

所有query一股脑丢进向量检索的后果：该算的不算，该过滤的不过滤。

2. 三级意图识别组合方案

  • 第一级：规则匹配——关键词能命中的直接走规则，零延迟
  • 第二级：BERT分类器——规则匹配不上的，几十毫秒解决
  • 第三级：LLM兜底——softmax最高类别概率仅0.4时才调LLM

路由逻辑：知识问答 → 混合检索 | 计算求解 → 计算模块 | 数据查询 → NL2SQL | 闲聊 → LLM通用对话

进阶策略：多索引路由——知识库按主题分索引（理赔制度/销售策略/产品信息），意图识别后选对应索引检索，既提精度又减计算量。

四、生成阶段：不是检索+生成的简单加法

1. Prompt约束：让LLM闭嘴只说文档里有的

最初Prompt只是「根据以下文档回答问题」，LLM经常自己发挥——文档明明说等待期90天，它非要补充「建议咨询客服」。

改为强约束Prompt的四条规则：

  1. 只使用文档中的信息，不要添加额外推理
  2. 文档中没有明确答案时回复「知识库中暂无相关信息」
  3. 不要说「可能」「大概」「建议咨询」等模糊表达
  4. 引用文档时注明来源（文档名称+章节）

2. 置信度评估：让LLM知道自己不知道

让LLM在答案末尾输出置信度分数（1-5分）。低于3分时自动触发二次检索——用改写后的query重新搜索，或降低召回阈值扩大范围。

「答非所问」投诉率从 18% → 7%

五、系统协同：RAG最难的部分

每个模块各司其职不难，难的是让它们协同到最优——需要既懂算法又懂工程。

  • 文档更新——知识库不是一次性导入。业务文档会更新，旧版本标记失效，新版本及时入库。方案：每天凌晨跑增量更新脚本，只对变化部分重新embedding。
  • 向量检索性能——5000份还好，50万份呢？用Milvus分区索引，按业务线分分区，检索时只搜相关分区，速度提升3倍。
  • Prompt格式——5个chunk平铺会让LLM被前面内容带偏。做法：给每个chunk加序号和来源标签，让LLM知道这是5段独立信息而非连贯文章。
  • 响应速度——两层优化：高频query做缓存（同问题直接返回）；流式输出（生成一句推一句）。

六、面试怎么聊RAG难点

  1. 系统性思维——RAG不是单点算法，是端到端系统工程，环环相扣，任何一环瑕疵都导致答案质量下降。
  2. 具体的坑——切块策略、召回阈值调优、意图识别、Prompt约束，每个环节都有技术决策和权衡。
  3. 量化指标——不说「优化了检索」，说「召回准确率0.62→0.91」；不说「改进了Prompt」，说「投诉率18%→7%」。
  4. 工程能力——文档更新、性能优化、缓存策略、监控告警，这些工程细节才是面试官最想听的。
