一、为什么需要意图识别？

AI以接收用户自然语言为交互基础，首先要理解用户到底想做什么——理解需求、对齐目标，AI才不至于跑偏。意图识别错误会导致整个AI处理流程错误。

意图识别的核心任务：准确判断用户目的，将用户输入映射到预定义的意图类别。

ChatBI场景下的意图识别还需额外解析：从知识库检索指标定义/描述、识别指标名称、确定指标相关维度信息。

二、意图分类：六类预定义意图

严格限定AI只处理指定业务范围内明确规定的流程，避免偏离用户真实需求。ChatBI场景下的预定义分类：

类别说明典型输入
QUERY查询类：询问指标、维度、含义、数据、业务情况"华东区Q4销售额是多少"
HI寒暄类：打招呼、自我介绍、问你是谁"你好，你是什么模型"
SUMMARY总结类：对前文做小结、通过前文总结回答问题"总结一下刚才的分析"
UNRELATED无关类：与业务范围完全无关"今天天气怎么样"
FILE文件类：上传、更新、替换数据文件"上传这份Excel"
ASK反问类：无法判断分类，需向用户追问"帮我分析一下"（分析什么？）

三、意图识别基本概念

1. 流程中的节点

意图识别在整个ChatBI流程中的位置：

用户输入 → 意图识别Agent → 应用路由 → SQL生成Agent → SQL执行 → 返回结果

2. 槽位（Slot）概念

槽位是意图识别中的核心概念，用于从用户输入中提取具体参数/信息：

概念比喻例子
意图(Intent)你想做什么"买机票"
槽位(Slot)做这件事需要什么信息出发地、目的地、日期、人数
槽位值具体的信息北京、上海、2025-04-01、2人

3. 槽位类型

  • 必填槽位——必须有值才能执行，如区域、时间
  • 可选槽位——有值时使用，没有时用默认值，如指标
  • 枚举槽位——只接受预定义值，如指标类型：销售额/订单数
  • 时间槽位——日期、月份、季度等
  • 实体槽位——商品名、地名、人名等实体

4. 槽位处理流程

以"分析华东区的销售数据"为例：

  1. 用户输入："分析华东区的销售数据"
  2. 匹配意图：销售数据分析；提取槽位：区域=华东区✓，时间=未找到✗，指标=销售✓
  3. 检查必要槽位：区域有值✓，时间为空✗（缺少），指标有值✓
  4. 缺少必要槽位 → 向用户提问："请提供分析的时间范围，如Q4或具体月份"
  5. 用户补充："是2025年Q4"
  6. 最终完整槽位：{ 区域: 华东区, 时间: 2025年Q4, 指标: 销售 }

四、大模型实现意图识别的流程

通过大模型实现意图识别与槽位抽取，需要做好两件事：

  1. 定义意图槽位——让大模型精确理解每个意图的范畴，以及对应槽位的名称、数据类型与取值范围
  2. 为每个意图搭配示例——提供典型范例，让大模型理解如何从输入文本精准抽取槽位信息

各意图槽位定义

  • QUERY 槽位：查询目标、数据范围、业务场景
  • HI 槽位：无
  • SUMMARY 槽位：总结范围、总结粒度
  • UNRELATED 槽位：误匹配原因
  • FILE 槽位：文件操作类型、文件类型、文件描述
  • ASK 槽位：缺失信息类型、追问内容

识别流程三步走

  1. 构建Prompt（包含：预定义意图列表、每个意图的槽位定义、用户输入、输出格式要求）
  2. 调用LLM进行意图匹配
  3. 解析LLM输出 → 意图 + 槽位 + 补充问题

五、Prompt设计：意图识别的核心

1. 核心要点

  • 明确各类意图及槽位定义——对应槽位的名称、数据类型与取值范围，打造详尽的任务字典
  • 为每个意图提供示例——典型范例让模型领会如何精准抽取槽位信息
  • 定义输出格式——以结构化JSON输出，确保后续系统便捷解析

2. Prompt模板结构

模板包含六个模块：

  1. 角色定义——"你是意图识别专家"
  2. 目标——从意图定义中选择最匹配的意图并提取槽位
  3. 约束规则——5条核心约束：严格按定义输出、从历史对话提取槽位、空值处理、不填无效信息、缺失槽位时生成追问
  4. 响应格式——JSON格式：{intent, slots, ask_user, user_input}
  5. 示例——典型输入输出对
  6. 意图槽位定义——当前任务的完整意图字典

3. 五条约束规则详解

  1. 严格根据给出的意图定义输出，不要自行生成意图和槽位属性
  2. 从用户输入和历史对话中提取槽位值，无法获取则输出空
  3. 槽位值只获取有效值部分，不要填入辅助描述或定语；未找到值的槽位也要输出属性名和空值
  4. 未提供的槽位值必须为空，不要填"用户未提供"等无效信息
  5. 槽位无法完全对应时，生成追问问题提示用户补充

六、优化方案：意图向量化

直接用大模型做意图识别的问题：意图数据量多时，为每个意图定义槽位和示例会让Prompt膨胀，模型难以准确捕捉关键内容，容易混淆和错误关联。

核心矛盾：意图越多 → Prompt越长 → 模型理解越困难 → 识别准确率下降

解决方案：把意图向量化——

  1. 在向量库里上传大量意图分类知识
  2. 用户提问时，先到向量库查询和用户请求类似的知识
  3. 把召回的知识和用户请求一起输入给大模型
  4. 大模型只在少量相关意图中做识别
效果：减少Prompt内容（只传最相关的意图而非全部意图），提升识别准确率，降低Token消耗。
