AI时代下数据开发工作方式转变

从传统数开到AI+数据开发的技能树重构与实践路径
原文来源语兴

一、AI时代背景与模型格局

1. 当前AI发展态势

随着Claude Opus 4.6、GPT 5.4、Gemini 3.1 Pro、GLM 5等模型日趋成熟，加上Open Claw、Skill、MCP等新范式出现，AI正在重构所有行业的生产方式：

  • 大厂裁员≠AI淘汰程序员，而是工作模式变了——开发效率/产出效率极大提升
  • AI影响所有行业：音乐创作被SUNO V5替代，短视频可用Seedance 2.0生成
  • 未来考察人才最重要的是Idea（想法），而不是实现方式
  • 有了好Idea，AI就能帮你开发——工具、网站、脚本都能实现
核心观点：提示词 + 架构思维 + 创新能力 = 未来招聘要求

2. 大模型能力排名（推荐优先级）

排名模型适用场景
🥇Claude Opus 4.6程序开发、文档编写、复杂改造（最推荐）
🥈GPT 5.4日常问答、报错分析、通用任务
🥉Gemini / Grok多模态、长上下文任务
4GLM 5国产替代、成本敏感场景
5Kimi长文档阅读、分析
6Qwen / Minimax / 豆包轻量任务、国内合规
注意：很多公司给员工用一般模型还要控额度去阉割模型，反而没提效。Claude Opus 能够完全替代程序员进行业务程序开发及文档编写，只是费 Token。

二、AI + 数据开发的日常实践

1. 日常问答：报错与倾斜分析

报错分析：将报错信息和数据代码直接给 GPT/Codex/GLM/Kimi 即可

  • 数据倾斜 → 推荐 Claude Opus / GPT，需配合 Spark Web UI 截图
  • Python/Java 开发 → 不需要你开发，但需要设计方案，用 Claude Opus / GPT
  • 好的提示词 = 需求描述 + 表结构 + 上下文信息 + 开发规范
关键洞察：在AI时代，会写提示词的同学非常占优——有理科的架构思维 + 文科的内容总结，才是用好AI的关键。

2. 工具开发（两种类型）

类型一：开发平台类工具——取数平台、指标平台、ChatBI、监控平台、工单回复等

  • 需要写好 Prompt + PRD 文档，让 AI 读取
  • 想好技术栈、申请域名、UI 设计
  • 通过 Vibe Coding 快速实现，例如一个平台从 PRD 设计到发布只需 3 天
  • 没有原型图？去官网买一个月玩玩，或直接截图喂大模型

类型二：解决数开痛点工具——同步工具（飞书2Hive）、DQC 自增等

  • 发现业务及组内痛点，快速用 Claude Opus 开发 Python 脚本解决
  • 代码不是壁垒，好的 Idea 才是

三、Dify + 数开组合

1. Dify 平台核心功能

功能模块说明
探索可查看和使用定义好的 Chatbot、Chatflow、Workflow、Agent
工作室可查看、编辑和创建 Chatbot、Chatflow、Workflow、Agent
知识库管理知识库，作为上下文被其他智能体引用
工具集成外部工具和 API

2. 数据工作流

前置数据推送：清洗好原始宽表数据，选择贴合场景的内容数据

  • 风控场景 → 爬取擦边/涉政内容
  • 优质推荐 → 爬取优质内容
  • 数仓链路：ODS（接入原始数据）→ DWD（JSON解析/清洗）→ ADS（打标/封装）
  • 最终封装 JSON 推送大模型平台

后置数据测算：Workflow 产出数据评估业务召回/准确率

  • 结合多维度属性调整大模型 Prompt
  • 构建 ODS→ADS 链路，建设指标、搭建报表
模型选型建议：Dify 中建议用便宜模型（GPT 5.2、GLM、Kimi），因为数据是单条处理，非常消耗 Token。

四、Skill + Claw + 数开组合

1. 核心概念

  • Agent（智能体）：能感知环境、决策和执行动作的实体，有记忆、逻辑分析、任务拆解和综合解决问题的能力
  • Skill：就像离线任务封装的代码，逻辑可复用
  • Claw：就像调度器（类比 Dolphin Scheduler）

2. Skill 目录结构

.claude/skills/my-skill/
├── SKILL.md              # 核心指令文件（必须）
├── template.yaml          # 可选模板文件
├── scripts/
│   └── validate.sh       # 可选脚本
└── examples/              # 可选参考示例

3. Skill 触发方式

  • 手动触发：用户调用 /skill-name [arguments]
  • 自动触发：Claude 分析用户请求，匹配 description 字段自动调用

4. 应用场景

  • 任务报错分析：给出原因和修改建议
  • 数据治理建议：让 Skill 给建议，人工确认后再执行
  • 省钱组合策略：用 Claude Opus 4.6 开发 Skill，日常用 GPT 5.2 / GLM / Kimi 调用分析

五、传统 RAG 与数据开发

1. RAG 本质

将长文本进行 Split 分块，搜索到的资料统一发给知识库检索，将查到的内容融合提示词发给大模型。

痛点：跨页映射信息容易检索不准（可能跨了好几个文档），Context Rot 问题——随着 Prompt/上下文变长，LLM 效果下降。

2. 数开场景下的 RAG 应用

  • 大模型 + 数据问答助手：前端 AI 生成，后端 Dify 平台
  • 读取核心表数据集及文档
  • 温度设置：
    • 咨询查表 → 低温（<1），避免天马行空
    • 财报/报表分析 → 高温（>1.5），需要创造性
  • 前置判断用户目的：可查阅的给文档，定期清理过时内容和数据集信息

六、MCP + 数开组合

1. MCP 是什么

MCP 是大厂数据平台在 2026 的重要方向——通过 Skill.md 实现需求开发，未来数据开发交互只需和大模型对话即可完成数据开发及数据平台功能交互。

2. MCP 配置流程

  1. 在平台生成 MCP Token 的 ID
  2. 新建终端 cd ~，vim .claude.json，:wq 保存
  3. 重启 VS Code
  4. 写提示词调用 MCP 服务工具（如 get_table_ddl 查询表结构）

七、数据安全与合规

  • 账号/密码、MySQL 实例等信息切勿放到公网，询问时可置空
  • 数据报告生成建议使用内部 ChatBI 进行
  • 报告信息切勿透露公司信息
  • 如果不方便投喂数据，让大模型提供专题分析报告模板，人工填指标数据

八、总结与未来方向

公式：提示词 + 架构思维 + 创新能力 = 未来招聘要求
  • AI 在生产提效方面确实是强力 Buff
  • 混沌期是转型大模型应用的最佳时机——还没有明确的招聘标准
  • 无论之前是数分、数开，未来可能只有数据岗这一种岗位了
  • 简历上有 AI 项目会优先考虑
  • SaaS 未来会很难，得找到自己的竞争优势

普通人会大模型应用基本就够了——如同会开车和研究汽车引擎的区别。
